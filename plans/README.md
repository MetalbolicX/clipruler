Generated by the `improve` skill on 2026-07-20. Execute in the order below unless dependencies say

your row when done.

## Cross-cutting policies (apply to every plan)

### Dependency policy — std first

- **Default to Deno standard library** (`jsr:@std/*`) and **WebCrypto**. Any third-party dependency
  requires an explicit ROI/tech-debt note inside the plan that uses it, covering: maintainer health,
  alternatives, blast radius if it disappears, and rewrite cost.
- **Std picks for v0.1.0** (no justification needed): `@std/cli`, `@std/path`, `@std/fs`,
  `@std/async`, `@std/crypto`, `@std/encoding`, `@std/uuid`, `@std/testing`, `@std/log`,
  `@std/bytes`, `@std/streams`, `@std/semver`, WebCrypto (`crypto.subtle`) for Ed25519.
- **Deferred / banned without explicit ADR**: mDNS libraries (UDP beacon in MVP), TLS cert libraries
  (manual ASN.1 DER or `Deno.createServer` with self-signed), full CLI frameworks (Cliffy), schema
  validators (Zod), bcrypt or Argon2 (SHA-256 truncation is enough for a local pairing code).

### Architecture policy — Hexagonal

- `src/domain/` — pure TypeScript, no I/O, no `Deno.*` except types.
- `src/ports/` — interfaces only.
- `src/application/` — use cases that orchestrate ports + domain.
- `src/infrastructure/` — adapters implementing ports.
- `src/protocol/` — wire format, independent and versioned.
- `src/shells/` — composition roots; only place that knows concrete adapters.
- Dependency direction: `shells → application → ports ← infrastructure`, `application → domain`.
  Never `domain → *`.

### Verification policy

Every plan ends with: `deno fmt --check`, `deno lint`, `deno check`, and `deno test` all green.
Tests in `domain/` MUST NOT import `Deno.*` runtime symbols (type-only imports allowed) so they
remain deterministic.

## Dependency notes

- 005 and 006 can be done in parallel after 004 if two executors are available.
- 007 is the largest single phase; it wires pairing state machine, code derivation, persistence, and
  rejection of untrusted peers.
- 011 depends on 010 because the desktop shell reuses the admin channel for in-process IPC between
  the webview and the daemon core.
- 012 should be executed last; packaging requires the full module graph.

## Findings considered and rejected

- mDNS via third-party library: rejected for v0.1.0; UDP beacon is enough for LAN discovery and
  avoids a non-std dependency.
- macOS support in v0.1.0: deferred; the adapter is trivial (`pbcopy`/ `pbpaste`) but CI matrix and
  packaging need a separate effort.
- Images and files in the clipboard: deferred to v0.2; binary payloads would force transport
  changes.
- Native OS auto-start (systemd/LaunchAgent/Windows Service): deferred until the foreground daemon
  is stable.
